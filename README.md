> Built an AI-powered QA automation platform using **Claude Code multi-agent orchestration** — 4 specialized skill agents (scenario generation, test strategy, test generation, code review) that read domain knowledge, live source code, and business rules before producing output; generated **53 test scenarios stratified across a proper test pyramid** (unit / API / component / E2E) for a full-stack **Next.js + Express + MySQL** application.

## What is this?

Most AI-assisted testing tools generate generic tests in isolation. This project is different.

EventHub is a production-quality ticket booking application used as a live **system under test**. On top of it sits a **Claude Code multi-agent QA framework** where 4 specialized AI agents collaborate in sequence — each reading domain knowledge, business rules, and actual source code before producing any output.

The result: structured test scenarios, a rationalized test pyramid, self-validating Playwright tests, and AI-driven code review — all grounded in real application behavior, not guesswork.

## The 4 agents

```
.claude/skills/
├── create-scenarios/    AI Functional Tester    — generates 53 scenarios across 6 thinking lenses
├── test-strategy/       AI Test Architect       — assigns pyramid layers with source-traced rationale
├── generate-tests/      AI Test Engineer        — writes, runs, diagnoses, and fixes Playwright tests
└── review-tests/        AI Code Reviewer        — audits tests against standards, selectors, domain rules
```

### Agent pipeline

```
/create-scenarios <feature>
         │
         ▼
  docs/test-scenarios.md        53 scenarios · 6 categories · traced to business rules
         │
         ▼
/test-strategy <scenarios>
         │
         ▼
  docs/test-strategy.md         layer assignments · rationale · anti-pattern flags
         │
         ▼
/generate-tests <feature>
         │
         ├── reads domain knowledge + source code
         ├── writes test file
         ├── runs tests in real browser (Playwright MCP)
         ├── diagnoses failures: error message + live DOM + source code
         └── fixes and re-runs until green
         │
         ▼
/review-tests <file>
         │
         └── cross-references best practices · verifies selectors in source · checks domain assertions
```

## Test coverage produced

53 scenarios generated by `/create-scenarios`, stratified by `/test-strategy` into a proper test pyramid:

| Layer | Count | Focus |
|---|---|---|
| Unit | 5 | Pure functions — `randomRef()`, price calc, ref uniqueness retry |
| API / Integration | 22 | Backend contracts, auth enforcement, business rules, DB state |
| Component | 14 | UI states, conditional rendering, mocked API responses |
| E2E | 12 | Critical user journeys, multi-page flows, cross-session security |
| **Total** | **53** | |

**Pyramid shape: Unit (5) → API (22) → Component (14) → E2E (12) ✓**

Scenarios follow a structured taxonomy:

| TC range | Category |
|---|---|
| TC-001–099 | Happy path |
| TC-100–199 | Business rules |
| TC-200–299 | Security |
| TC-300–399 | Negative / error |
| TC-400–499 | Edge cases |
| TC-500–599 | UI state |

### Defense-in-depth: critical rules covered at multiple layers

| Rule | Unit | API | Component | E2E |
|---|---|---|---|---|
| Booking ref prefix = event title's first character | ✓ | ✓ | — | ✓ |
| Total price = price × quantity | ✓ | ✓ | — | implicit |
| Refund eligible only for qty = 1 | — | — | ✓ | — |
| Cross-user booking access returns 403 | — | ✓ | — | ✓ |
| Cancel booking: data deleted, redirect shown | — | ✓ | — | ✓ |
| FIFO pruning at 9-booking limit | — | ✓ | — | — |
| Auth required on all booking endpoints | — | ✓ | — | — |

### Anti-patterns the `/test-strategy` agent flagged and avoided

| Anti-pattern | Correct layer instead |
|---|---|
| Testing input validation (qty = 0, qty > 10) at E2E | API — `POST /api/bookings` → assert 400 |
| Testing 4-second refund spinner at E2E (login + navigate + wait) | Component — render with mocked `quantity` prop |
| Testing FIFO pruning at E2E (requires 9 pre-seeded bookings) | API — precise DB setup per test |
| Testing "Booking not found" by navigating to `/bookings/99999` at E2E | Component — mock `isError=true` via route interception |
| Testing 401 responses by manipulating UI session | API — send requests without `Authorization` header |

## The system under test

EventHub is a real, production-quality ticket booking application. It gives the agents meaningful complexity: layered architecture, business rules with edge cases, security boundaries, and client-side state machines.

### Tech stack

| Layer | Technology |
|---|---|
| Frontend | Next.js 14 (App Router), React 18, TypeScript, Tailwind CSS, React Query v5 |
| Backend | Node.js, Express.js, Prisma ORM, Swagger UI |
| Database | MySQL 8 |
| Auth | JWT (7-day expiry), bcryptjs |
| Testing | Playwright E2E — Chromium |

### Business rules baked into the agents' domain knowledge

- **Booking limit**: max 9 per user — FIFO pruning on overflow; oldest booking deleted first, with same-event fallback that permanently burns a seat
- **Booking reference format**: `[FIRST_CHAR_OF_EVENT_TITLE]-[6_RANDOM_ALPHANUM]` — validated at unit + API + E2E layers for defense-in-depth
- **Seat atomicity**: seat count decrements on booking, restores on cancellation
- **Refund eligibility**: client-side only — qty = 1 is eligible, qty > 1 is not; 4-second simulated check with spinner state machine
- **Security boundary**: cross-user booking access returns `403 Access Denied` — enforced at service layer, validated at API and E2E layers
- **Event limit**: max 6 user-created events — FIFO pruning on overflow
- **Static events**: seeded events are immutable — cannot be edited or deleted

## Running locally

### Prerequisites

- Node.js 18+
- MySQL 8+

### Setup

```bash
# Clone and install
git clone <repo-url>
cd eventhub
npm run setup

# Create the database
mysql -u root -p -e "CREATE DATABASE eventhub CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# Configure environment
# backend/.env:
#   DATABASE_URL="mysql://root:your_password@localhost:3306/eventhub"
#   PORT=3001
#   NODE_ENV=development
#   CORS_ORIGIN=http://localhost:3000
#
# frontend/.env.local:
#   NEXT_PUBLIC_API_URL=http://localhost:3001/api

# Push schema and seed data
npm run db:push
npm run seed

# Start both servers
npm run dev
```

| Service | URL |
|---|---|
| Frontend | http://localhost:3000 |
| Backend API | http://localhost:3001 |
| Swagger UI | http://localhost:3001/api/docs |

### Running the tests

```bash
npm run test                                                          # all tests, headless
npm run test:ui                                                       # Playwright UI mode
npx playwright test tests/booking-management.spec.js --reporter=line # single file
```

## Using the agents

With [Claude Code](https://claude.ai/code) installed and this repo open:

```bash
/create-scenarios booking-flow      # generate structured test scenarios
/test-strategy booking              # assign pyramid layers with rationale
/generate-tests booking-flow        # write, run, debug, and fix Playwright tests
/review-tests tests/booking-management.spec.js   # audit against standards and domain rules
```

Each agent reads its full knowledge chain before producing a single line of output.
